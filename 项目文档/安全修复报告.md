### SEC-002: 登录防暴力破解

#### 问题描述
登录接口 `/api/auth/login` 没有任何限流措施，攻击者可以无限尝试密码。

#### 修复方案
1. **全局限流保护**：
   - 在 `AppModule` 中配置 `ThrottlerModule`
   - 添加全局 `ThrottlerGuard`

2. **登录专用限流**：
   - 创建 `LoginThrottlerGuard` 自定义限流守卫
   - 登录接口配置：`5次/15分钟`

3. **账户锁定机制**：
   - 登录失败 5 次后自动锁定 15 分钟
   - 实现 `recordLoginFailure()`, `lockAccount()`, `isAccountLocked()` 方法
   - 登录成功自动清除失败记录

4. **友好错误提示**：
   - 显示剩余尝试次数
   - 显示锁定剩余时间

#### 修改文件
- `src/app.module.ts` - 添加 ThrottlerModule
- `src/common/guards/login-throttler.guard.ts` - 新建
- `src/modules/user/user.controller.ts` - 添加限流装饰器
- `src/modules/user/user.service.ts` - 添加锁定逻辑

#### 验证方法
```bash

curl -X POST http://localhost:3001/api/auth/login \
  -H "Content-Type: application/json" \

# 2. 连续5次失败后被锁定
# 预期返回: 账户已被锁定，请在 15 分钟后重试

# 3. 15分钟后自动解锁（内存缓存，生产环境建议用Redis）
```

### SEC-003: 注册机器人验证

#### 问题描述
注册接口 `/api/auth/register` 无任何验证机制，可被批量注册。

#### 修复方案
1. **新增图形验证码模块**：
   - 创建 `CaptchaModule` 包含服务、控制器
   - 支持 SVG 图形验证码生成
   - Token 机制防止重放攻击
   - 5分钟有效期

2. **注册接口集成**：
   - 添加 `captchaToken` 参数
   - 可通过配置 `ENABLE_CAPTCHA` 开关

3. **验证码API**：
   - `GET /api/captcha/image` - 获取验证码图片和Token
   - `POST /api/captcha/verify` - 验证验证码

#### 修改文件
- `src/modules/captcha/captcha.service.ts` - 新建
- `src/modules/captcha/captcha.controller.ts` - 新建
- `src/modules/captcha/captcha.module.ts` - 新建
- `src/modules/user/user.controller.ts` - 集成验证码验证
- `src/app.module.ts` - 引入 CaptchaModule

#### 验证方法
```bash
# 1. 获取验证码
curl -X GET http://localhost:3001/api/captcha/image \
  -i
# 响应头包含: X-Captcha-Token: xxx

# 2. 使用验证码注册
curl -X POST http://localhost:3001/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"name":"Test","email":"test@example.com","password":"password123","captchaToken":"xxx"}'
```

#### 备选方案：集成 reCAPTCHA
如需更强的验证码，可配置第三方 reCAPTCHA：
```bash

RECAPTCHA_SECRET_KEY="your-secret-key"
```

### SEC-004: 资源隔离不完整

#### 问题描述
订单更新接口 `/api/orders/:id/status` 未验证订单所属用户，任何登录用户可修改他人订单。

#### 修复方案
1. **控制器层修复**：
   - `updateStatus()` 方法添加 `req.user.id` 验证
   - 查询订单时必须匹配 `userId`
   - 不匹配返回 403 Forbidden

2. **服务层修复**：
   - `updateStatus()` 方法强制要求 `userId` 参数
   - 查询前先验证订单归属

3. **增强验证**：
   - 添加 `ForbiddenException` 异常处理
   - 状态值白名单验证

4. **新增安全端点**：
   - 添加 `POST /api/orders/:id/cancel` 订单取消接口
   - 同样包含完整的用户验证

#### 修改文件
- `src/modules/order/order.controller.ts` - 添加用户验证
- `src/modules/order/order.service.ts` - 修复 updateStatus 方法

#### 验证方法
```bash
# 1. 用户A创建订单
curl -X POST http://localhost:3001/api/orders \
  -H "Authorization: Bearer [UserA_Token]" \
  -H "Content-Type: application/json" \
  -d '{"totalAmount":100,"items":[]}'

# 2. 用户B尝试修改用户A的订单
curl -X PUT http://localhost:3001/api/orders/[OrderA_ID]/status \
  -H "Authorization: Bearer [UserB_Token]" \
  -H "Content-Type: application/json" \
  -d '{"status":"shipped"}'

# 预期返回: 403 Forbidden {"message":"无权操作此订单"}
```
