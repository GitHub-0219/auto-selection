# 质量管理制度

## 一、代码质量标准

### 1.1 代码规范
- **语言规范**：遵循业界通用规范（如Google Style Guides）
- **命名规范**：
  - 变量/函数：驼峰命名（camelCase）
  - 类名：大驼峰（PascalCase）
  - 常量：全大写下划线分隔（CONSTANT_CASE）
- **注释规范**：
  - 公共API必须注释
  - 复杂逻辑必须注释
  - 保持注释与代码同步

### 1.2 代码审查（Code Review）
| 要求 | 说明 |
|-----|-----|
| 强制要求 | 所有代码合并前必须经过Review |
| Reviewer | 至少1人，需是同模块或相关模块开发者 |
| 审查重点 | 代码逻辑、命名规范、安全性、性能、可读性 |
| 通过标准 | 至少1人Approve，无阻塞性问题 |
| 时效要求 | 提交后24小时内完成Review |

### 1.3 代码审查清单
```
□ 代码逻辑正确性
□ 边界条件处理
□ 错误处理是否完善
□ 命名是否清晰易懂
□ 是否有重复代码（DRY原则）
□ 是否有安全漏洞（SQL注入、XSS等）
□ 是否有性能问题（大循环、N+1查询等）
□ 测试覆盖是否充分
□ 文档/注释是否完整
□ 是否符合编码规范
```

### 1.4 技术债务管理
- **定义**：为快速交付而做出的非最优技术选择
- **管理原则**：
  - 允许存在，但必须记录
  - 定期偿还，不得无限累积
  - 新增债务需评估ROI
- **处理方式**：
  - 创建技术债务Ticket
  - 纳入迭代计划
  - 定期专项处理

### 1.5 分支管理策略（Git Flow简化版）
```
main (生产环境)
├── develop (开发主分支)
│   ├── feature/xxx (功能分支)
│   ├── bugfix/xxx (Bug修复分支)
│   └── release/xxx (发布分支)
```

| 分支类型 | 命名规范 | 合并规则 |
|---------|---------|---------|
| feature | feature/功能名-日期 | PR → develop |
| bugfix | bugfix/问题名-日期 | PR → develop |
| release | release/v版本号 | PR → develop + main |
| hotfix | hotfix/问题名-日期 | PR → main + develop |

## 二、测试规范（参考Google测试文化）

### 2.1 测试理念
- **质量是每个人的责任**：开发不是把代码"扔"给测试
- **左移测试**：尽早发现问题，越早修复成本越低
- **自动化优先**：重复的手工测试必须自动化
- **测试金字塔**：
  - 单元测试（70%）：快速反馈，定位准确
  - 集成测试（20%）：验证模块间协作
  - 端到端测试（10%）：验证完整用户场景

### 2.2 测试覆盖率要求
| 代码类型 | 最低覆盖率 | 目标覆盖率 |
|---------|----------|----------|
| 核心业务逻辑 | 80% | 90% |
| 工具类/公共方法 | 70% | 85% |
| 新增代码 | 80% | 90% |
| Bug修复 | 必须有测试 | 防止回归 |

### 2.3 测试类型与规范
| 测试类型 | 英文名 | 执行频率 | 责任人 |
|---------|-------|---------|-------|
| 单元测试 | Unit Test | 每次提交 | 开发 |
| 集成测试 | Integration Test | 每日 | 开发 |
| 端到端测试 | E2E Test | 每次发布前 | QA/开发 |
| 冒烟测试 | Smoke Test | 每次发布后 | QA |

### 2.4 单元测试规范
```python
# 测试文件命名：xxx_test.py
# 测试类命名：TestXxx
# 测试方法命名：test_xxx_should_xxx

class TestUserService:
    def test_create_user_should_return_user(self):
        # Arrange - 准备测试数据
        user_data = {"name": "Test", "email": "test@example.com"}
        
        # Act - 执行被测试的方法
        result = user_service.create(user_data)
        
        # Assert - 验证结果
        assert result.name == "Test"
        assert result.email == "test@example.com"
    
    def test_create_user_with_invalid_email_should_raise_error(self):
        # 负面测试：验证异常情况
        with pytest.raises(InvalidEmailError):
            user_service.create({"email": "invalid"})
```

### 2.5 测试用例设计原则
| 原则 | 说明 | 示例 |
|-----|-----|-----|
| 等价类划分 | 同类输入只测一个 | 1-100视为一类 |
| 边界值分析 | 重点测试边界 | 0、1、99、100、101 |
| 正反测试 | 既测正确也测错误 | 合法邮箱 + 非法邮箱 |
| 独立性 | 测试间无依赖 | 每个测试独立运行 |
| 可重复性 | 相同结果可重复 | 不依赖外部状态 |

### 2.6 自动化测试要求
| 类型 | 阈值 | 卡点 |
|-----|-----|-----|
| 单元测试覆盖率 | ≥70% | CI阻断 |
| 集成测试 | 核心流程全覆盖 | CI阻断 |
| E2E测试 | 关键路径覆盖 | 手动验证 |

## 三、发布流程

### 3.1 发布策略
| 环境 | 用途 | 更新频率 | 负责人 |
|-----|-----|---------|-------|
| 开发环境 | 开发调试 | 随时 | 开发 |
| 测试环境 | QA测试 | 每日构建 | CI/CD |
| 预发布环境 | 上线前验证 | 每周 | QA+产品 |
| 生产环境 | 正式服务 | 按发布计划 | 运维 |

### 3.2 发布类型
| 类型 | 说明 | 提前通知 | 审批人 |
|-----|-----|---------|-------|
| 常规发布 | 功能迭代、Bug修复 | 2天 | 部门负责人 |
| 紧急发布 | 重大Bug修复、安全漏洞 | 即时通知 | CEO |
| 热更新 | 不重启服务的小更新 | 1天 | 技术负责人 |

### 3.3 发布流程（常规发布）
```
发布计划制定 → 代码冻结 → 测试验证 → 发布审批 → 执行发布 → 验证监控 → 事后回顾
```

### 3.4 各阶段要求
| 阶段 | 时长 | 责任人 | 产出物 |
|-----|-----|-------|-------|
| 发布计划制定 | 2天前 | 产品+研发 | 发布计划文档 |
| 代码冻结 | 发布前1天18:00 | 所有人 | 无新代码合并 |
| 测试验证 | 冻结后至发布 | QA | 测试报告 |
| 发布审批 | 发布前 | 产品+技术+CEO | 审批通过 |
| 执行发布 | 按计划时间 | 运维/开发 | 发布记录 |
| 验证监控 | 发布后2小时 | 全值班 | 监控报告 |

### 3.5 发布检查清单
```
□ 代码已Review通过
□ 测试用例全部通过
□ 测试覆盖率达标
□ 上线文档已更新
□ 监控告警已配置
□ 回滚方案已准备
□ 值班人员已就位
□ 通知相关方
□ 回滚触发条件已确认
```

### 3.6 回滚机制
- **触发条件**：
  - 错误率上升>5%
  - P99延迟>2秒
  - 核心功能不可用
- **回滚决策**：5分钟内决定是否回滚
- **回滚执行**：10分钟内完成
- **回滚后**：立即启动问题排查

## 四、缺陷管理

### 4.1 缺陷分级（参考行业标准）
| 级别 | 名称 | 定义 | 响应时间 | 修复时间 |
|-----|-----|-----|---------|---------|
| P0 | 严重 | 核心功能不可用、数据丢失、严重安全漏洞 | 立即 | 4小时内 |
| P1 | 高 | 主要功能受影响、无替代方案 | 1小时 | 24小时内 |
| P2 | 中 | 功能部分受影响、有替代方案 | 4小时 | 72小时内 |
| P3 | 低 | 界面问题、体验优化、非功能需求 | 1天 | 下一版本 |

### 4.2 缺陷处理流程
```
缺陷发现 → 提交缺陷单 → 确认优先级 → 分配开发 → 修复验证 → 关闭
```

### 4.3 缺陷单模板
```markdown
## 缺陷标题
[模块名] 缺陷描述

## 基本信息
- 缺陷编号：
- 严重级别：
- 发现时间：
- 发现人：
- 负责开发：

## 复现步骤
1. 
2. 
3. 

## 预期行为
...

## 实际行为
...

## 环境信息
- 测试环境：
- 版本号：
- 浏览器/客户端：
- 相关日志：

## 附件
- 截图/录屏/日志
```

### 4.4 缺陷响应要求
| 级别 | 首次响应 | 给出修复方案 | 完成修复 |
|-----|---------|-------------|---------|
| P0 | 立即 | 1小时内 | 4小时内 |
| P1 | 1小时内 | 4小时内 | 24小时内 |
| P2 | 4小时内 | 24小时内 | 72小时内 |
| P3 | 1天内 | 2天内 | 下一版本 |

### 4.5 缺陷统计与分析
| 指标 | 定义 | 目标 |
|-----|-----|-----|
| 缺陷逃逸率 | 生产缺陷数/总缺陷数 | < 10% |
| 缺陷修复周期 | 从提交到关闭的平均时间 | P0<4h, P1<24h |
| 缺陷reopen率 | 被重新打开的缺陷比例 | < 5% |
| 缺陷密度 | 缺陷数/代码行数 | < 1/千行 |

## 六、质量保障活动

### 6.1 持续改进机制
| 活动 | 频率 | 参与人 | 产出物 |
|-----|-----|-------|-------|
| Bug Bash | 每次大版本发布前 | 全员 | Bug清单 |
| 缺陷分析会 | 每月 | 开发+QA | 分析报告 |
| 质量回顾 | 每季度 | 技术团队 | 改进计划 |
| 技术优化专项 | 按需 | 相关开发 | 优化成果 |

### 6.2 质量文化建设（参考Google工程师文化）
- **学习分享**：每周技术分享会
- **案例复盘**：重大故障/问题必复盘
- **最佳实践**：沉淀推广成功经验
- **质量表彰**：对质量贡献者表彰

### 6.3 复盘机制（Post-mortem）
- **触发条件**：P0/P1缺陷、重大故障、客户投诉
- **目的**：不追责，重在学习
- **模板**：
  ```
  1. 事件概述
  2. 时间线
  3. 根因分析（5Why）
  4. 影响评估
  5. 改进措施（Action Items）
  6. 经验沉淀
  ```
