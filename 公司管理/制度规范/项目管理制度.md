# 项目管理制度

## 一、项目生命周期

### 1.1 项目定义
项目是为创造独特产品、服务或成果而进行的临时性工作。
- **临时性**：有明确的开始和结束
- **独特性**：产出是独特的
- **渐进性**：逐步完善

### 1.2 项目分类
| 类型 | 说明 | 典型周期 | 负责人 |
|-----|-----|---------|-------|
| 产品项目 | 新产品/功能开发 | 1-3个月 | 产品经理 |
| 技术项目 | 技术架构、基础设施 | 1-6个月 | 技术负责人 |
| 运营项目 | 运营活动、用户增长 | 2周-2个月 | 运营负责人 |
| 客户项目 | 定制化交付 | 按合同 | 项目经理 |

### 1.3 项目生命周期阶段
```
启动 → 规划 → 执行 → 监控 → 收尾
```

| 阶段 | 关键活动 | 产出物 | 评审点 |
|-----|--------|-------|-------|
| 启动 | 立项调研、目标确认、团队组建 | 项目章程、团队分工 | 立项评审 |
| 规划 | 范围定义、计划制定、风险识别 | WBS、进度计划、风险清单 | 规划评审 |
| 执行 | 任务实施、团队协作、交付物完成 | 可交付成果 | 周例会同步 |
| 监控 | 进度跟踪、质量控制、变更管理 | 状态报告、问题清单 | 周/双周评审 |
| 收尾 | 验收交付、复盘总结、经验沉淀 | 总结报告、资产归档 | 复盘评审 |

## 二、项目立项（参考敏捷开发理念）

### 2.1 立项原则
- **价值驱动**：优先做价值明确、ROI高的事
- **小步快跑**：大项目拆分为小迭代
- **快速验证**：先MVP验证，再迭代完善
- **跨职能协作**：产品、技术、运营共同参与

### 2.2 立项流程
```
需求提出 → 价值评估 → 优先级排序 → 立项评审 → 项目启动
```

### 2.3 需求来源
| 来源 | 说明 | 入口 |
|-----|-----|-----|
| 客户反馈 | 来自用户的实际需求 | 客服/反馈系统 |
| 市场调研 | 行业趋势、竞品分析 | 产品规划 |
| 数据驱动 | 数据分析发现的机会 | 数据团队 |
| 战略规划 | 公司战略分解 | CEO/管理层 |
| 技术驱动 | 技术创新带来的可能 | 技术团队 |

### 2.4 需求价值评估
| 维度 | 权重 | 评分标准 |
|-----|-----|---------|
| 用户价值 | 30% | 1-5分，解决问题的迫切程度 |
| 商业价值 | 30% | 1-5分，收入/用户增长潜力 |
| 技术可行性 | 20% | 1-5分，技术实现难度 |
| 市场时机 | 10% | 1-5分，是否最佳时机 |
| 战略契合 | 10% | 1-5分，与公司战略一致性 |

**优先级计算公式**：
```
综合得分 = 用户价值×0.3 + 商业价值×0.3 + 技术可行性×0.2 + 市场时机×0.1 + 战略契合×0.1
```

### 2.5 立项评审
| 评审内容 | 评审标准 | 评审人 |
|---------|---------|-------|
| 需求合理性 | 需求真实、痛点明确 | 产品 |
| 技术方案 | 方案可行、风险可控 | 技术 |
| 资源需求 | 人力、时间、成本合理 | 管理层 |
| 商业预期 | ROI合理、目标可达成 | CEO |

## 三、里程碑管理

### 3.1 里程碑定义
里程碑是项目中的重要检查点，标志着关键阶段的完成。

### 3.2 产品开发里程碑
| 里程碑 | 定义 | 评审标准 |
|-------|-----|---------|
| M1 概念验证 | 核心假设验证 | 用户需求确认、技术可行性验证 |
| M2 MVP完成 | 最小可行产品 | 核心功能可用、可用性达标 |
| M3 Beta发布 | 内部测试版本 | 功能基本完成、性能达标 |
| M4 正式发布 | 生产环境上线 | 质量达标、准备就绪 |
| M5 稳定运营 | 上线后稳定运行 | 指标正常、问题解决 |

### 3.3 里程碑评审会
| 评审项 | 说明 | 通过标准 |
|-------|-----|---------|
| 完成度 | 功能完成情况 | 核心功能100%，次要≥80% |
| 质量 | Bug数量、严重级别 | 无P0/P1，遗留P2<P5 |
| 性能 | 响应时间、并发 | 达到性能标准 |
| 文档 | 用户文档、技术文档 | 已完成 |
| 运营准备 | 运营物料、培训 | 已就绪 |

## 四、进度跟踪（参考敏捷开发）

### 4.1 敏捷实践
- **迭代周期**：2周一个Sprint
- **每日站会**：每天固定时间，15分钟内
- **Sprint规划**：每个迭代开始前
- **Sprint回顾**：每个迭代结束后
- **燃尽图**：每日跟踪进度

### 4.2 迭代管理流程
```
Sprint规划(周一) → 每日站会(每日) → Sprint评审(周五) → Sprint回顾(周五)
```

### 4.3 Sprint规划
| 环节 | 时长 | 内容 |
|-----|-----|-----|
| 上半场 | 1小时 | 选取Sprint待办项 |
| 下半场 | 1小时 | 任务分解、工作量评估 |

**Sprint目标制定**：
1. 回顾产品待办列表优先级
2. 选择承诺完成的待办项
3. 将待办项分解为任务
4. 团队确认可完成
5. 明确Sprint目标

### 4.4 每日站会（参考亚马逊"两个披萨原则"）
| 问题 | 回答 |
|-----|-----|
| 昨天做了什么？ | 完成的交付物 |
| 今天要做什么？ | 计划完成的工作 |
| 有什么阻碍？ | 需要帮助的问题 |

**规则**：
- 每人限时3分钟
- 只同步进展，不讨论细节
- 阻塞问题会后单独讨论
- 站立形式，简洁高效

### 4.5 进度可视化
| 工具 | 用途 |
|-----|-----|
| 看板 | 任务状态可视化 |
| 燃尽图 | 剩余工作量趋势 |
| Burnup图 | 已完成工作量趋势 |
| 累积流图 | 各状态任务分布 |

### 4.6 进度预警
| 预警级别 | 触发条件 | 处理措施 |
|---------|---------|---------|
| 🟡 黄色 | 进度延迟>20% | 增加资源、加班 |
| 🔴 红色 | 进度延迟>40%或关键路径延误 | 减少范围、延期或加资源 |

## 五、风险管理

### 5.1 风险识别
| 风险类别 | 常见风险 | 识别方法 |
|---------|---------|---------|
| 需求风险 | 需求变更、范围蔓延 | 需求评审、用户验证 |
| 技术风险 | 技术难点、系统稳定性 | 技术方案评审、架构设计 |
| 资源风险 | 人员变动、资源不足 | 资源规划、备份机制 |
| 进度风险 | 预估偏差、依赖延误 | 进度跟踪、关键路径 |
| 质量风险 | Bug过多、性能问题 | 测试覆盖、质量门禁 |
| 市场风险 | 竞品抢先、用户需求变化 | 市场监控、用户反馈 |

### 5.2 风险评估矩阵
|  | 影响低 | 影响中 | 影响高 |
|--|-------|-------|-------|
| **可能性高** | 🟡中风险 | 🔴高风险 | 🔴高风险 |
| **可能性中** | 🟢低风险 | 🟡中风险 | 🔴高风险 |
| **可能性低** | 🟢低风险 | 🟢低风险 | 🟡中风险 |

### 5.3 风险应对策略
| 策略 | 说明 | 适用场景 |
|-----|-----|---------|
| 规避 | 消除风险来源 | 高概率+高影响 |
| 转移 | 转移风险给他方 | 超出能力范围的风险 |
| 减轻 | 降低发生概率或影响 | 可控但无法消除 |
| 接受 | 接受风险，准备应急 | 低概率或低影响 |

### 5.4 风险跟踪
| 机制 | 频率 | 责任人 |
|-----|-----|-------|
| 风险清单 | 实时更新 | 项目经理 |
| 风险评审 | 周例会 | 项目团队 |
| 风险升级 | 立即 | 发现人→项目经理→CEO |

### 5.5 常见风险与应对
| 风险 | 预防措施 | 应对措施 |
|-----|---------|---------|
| 需求变更 | 充分需求确认、明确范围 | 评估影响、进入变更流程 |
| 技术难题 | 提前技术验证、请教专家 | 方案调整、引入外部资源 |
| 人员离职 | 知识文档、备份人员 | 快速招聘、工作交接 |
| 第三方依赖 | 多方案备选、合同约束 | 切换方案、加速沟通 |
| 进度延误 | 预留Buffer、分解任务 | 加班、删减范围、加资源 |

## 六、变更管理

### 6.1 变更分类
| 类型 | 定义 | 处理方式 |
|-----|-----|---------|
| 重大变更 | 影响范围>30%或延期>2周 | 重新立项评审 |
| 一般变更 | 影响范围10-30%或延期<2周 | 项目负责人审批 |
| 轻微变更 | 影响范围<10% | 记录即可 |

### 6.2 变更流程
```
变更提出 → 变更评估 → 变更审批 → 变更实施 → 变更验证
```

### 6.3 变更影响评估
| 评估维度 | 评估内容 |
|---------|---------|
| 范围影响 | 哪些功能受影响 |
| 进度影响 | 延期多少时间 |
| 成本影响 | 增加多少成本 |
| 质量影响 | 对质量的影响 |
| 资源影响 | 需要什么人 |

### 6.4 变更控制委员会（CAB）
- **成员**：CEO、产品负责人、技术负责人
- **职责**：审批重大变更
- **频率**：按需召开

## 七、跨部门协作

### 7.1 协作原则
- **共同目标**：以项目成功为导向
- **主动沟通**：提前沟通，不等问题发生
- **责任共担**：共同对结果负责
- **数据透明**：信息共享，状态可见

### 7.2 协作机制
| 机制 | 形式 | 频率 | 参与人 |
|-----|-----|-----|-------|
| 项目周会 | 线上会议 | 每周 | 项目全员 |
| 产品对齐会 | 线下/线上 | 每周 | 产品+技术+运营 |
| 跨部门同步会 | 线下/线上 | 每两周 | 各部门负责人 |
| 紧急协调会 | 按需 | 突发情况 | 相关方 |

### 7.3 问题升级机制
| 级别 | 定义 | 处理时间 | 升级路径 |
|-----|-----|---------|---------|
| L1 | 组内可解决 | 即时 | 组内解决 |
| L2 | 组间协调 | 4小时 | 项目经理协调 |
| L3 | 资源冲突 | 1天 | CEO协调 |
| L4 | 战略问题 | 3天 | 董事会 |

### 7.4 信息同步机制
| 信息类型 | 同步方式 | 同步频率 | 接收人 |
|---------|---------|---------|-------|
| 项目周报 | 邮件/文档 | 每周 | 项目干系人 |
| 里程碑状态 | 邮件 | 里程碑节点 | 管理层 |
| 风险预警 | 即时通讯 | 发生时 | 负责人 |
| 变更通知 | 邮件/会议 | 变更时 | 相关方 |

## 八、项目收尾

### 8.1 验收流程
```
交付物准备 → 验收测试 → 验收评审 → 正式上线 → 文档归档
```

### 8.2 验收标准
| 验收项 | 通过标准 | 验收人 |
|-------|---------|-------|
| 功能验收 | 所有功能按需求实现 | 产品负责人 |
| 性能验收 | 达到性能指标 | 技术负责人 |
| 文档验收 | 文档完整、更新 | 产品/技术 |
| 培训验收 | 相关人员完成培训 | 运营 |

### 8.3 复盘机制
| 复盘类型 | 时机 | 参与人 |
|---------|-----|-------|
| Sprint回顾 | 每个Sprint结束 | Sprint团队 |
| 项目复盘 | 项目结束2周内 | 项目全员 |
| 重大复盘 | 重大故障/失败 | 管理层参与 |

### 8.4 复盘模板
```markdown
# 项目复盘报告

## 基本信息
- 项目名称：
- 复盘时间：
- 参与人：

## 结果回顾
- 原定目标：
- 实际结果：
- 目标达成率：

## 过程分析
- 做得好的是什么？
- 做不好的是什么？
- 根因分析（5Why）：

## 经验沉淀
- 最佳实践：
- 教训总结：

## 改进行动
| 改进项 | 负责人 | 完成时间 |
|-------|-------|---------|
| | | |

## 待办事项
```
