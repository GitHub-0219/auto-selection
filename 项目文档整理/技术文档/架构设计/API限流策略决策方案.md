# API限流策略决策方案

**项目**：Auto选品（AI跨境电商SaaS）  
**版本**：v1.0  
**日期**：2026-04-10  
**目标用户规模**：首月50-100人

---

## 一、策略分析

### 1.1 两种模式对比

| 维度 | 保守模式（低QPS） | 激进模式（高QPS） |
|------|------------------|------------------|
| **API安全性** | ⭐⭐⭐⭐⭐ 稳定，不易触发第三方限流 | ⭐⭐ 风险较高，需完善重试机制 |
| **响应速度** | ⭐⭐ 高峰期可能较慢（1-3秒延迟） | ⭐⭐⭐⭐⭐ 快速响应（<500ms） |
| **开发复杂度** | ⭐⭐⭐⭐⭐ 简单，无需复杂重试逻辑 | ⭐⭐⭐ 需实现指数退避、熔断器 |
| **运维成本** | ⭐⭐⭐⭐⭐ 低，较少异常需处理 | ⭐⭐⭐ 需持续监控、调整阈值 |
| **用户满意度** | ⭐⭐⭐ 中等，可接受范围 | ⭐⭐⭐⭐⭐ 高，用户体验更好 |
| **成本控制** | ⭐⭐⭐⭐⭐ 更易控制API调用量 | ⭐⭐ 需更精细的配额管理 |

### 1.2 风险评估

**保守模式风险**：
- 高峰期用户等待时间较长
- 可能导致用户流失（特别是首批种子用户）
- 功能使用率下降，影响数据收集

**激进模式风险**：
- 第三方API限流触发（OpenAI/TikTok API等）
- API费用超出预算
- 服务不稳定影响口碑

---

## 二、推荐方案：渐进式激进模式

### 2.1 决策结论

**针对50-100人用户规模，推荐采用「渐进式激进模式」**：

> 核心思路：激进模式的高性能 + 保守模式的安全保障 + 智能降级机制

### 2.2 推荐理由

1. **用户规模适中**：50-100人不足以触发第三方主流限流（OpenAI默认60RPM，TikTok API通常>100RPM）
2. **首批用户珍贵**：种子用户对体验敏感度更高，应优先保证响应速度
3. **容错空间充足**：用户量小意味着有足够缓冲时间进行问题处理
4. **技术可行性**：当前团队技术能力可支撑重试+熔断机制的实现

## 三、限流算法选择

### 3.1 算法对比

| 算法 | 原理 | 优点 | 缺点 | 推荐场景 |
|------|------|------|------|----------|
| **令牌桶** | 桶内令牌决定请求通过 | 支持突发流量，内存占用低 | 实现稍复杂 | ✅ **首选** |
| **漏桶** | 固定速率消费请求 | 流量平滑，严格限流 | 无法处理突发 | 支付场景 |
| **滑动窗口** | 统计时间窗口内请求数 | 实现简单，精度高 | 内存占用随并发增长 | 简单限流 |

### 3.2 推荐：令牌桶算法

**原因**：
- TikTok数据查询具有「批量+突发」特性，令牌桶可吸收峰值
- Redis支持原生令牌桶（`INCR` + `EXPIRE`），实现简单
- 内存占用O(1)，适合微服务架构

## 四、具体配置参数

### 4.1 全局限流配置

```yaml
rate_limiting:
  # 全局入口限流（所有API请求）
  global:
    algorithm: "token_bucket"
    capacity: 100              # 令牌桶容量
    refill_rate: 50             # 每秒补充令牌数
    timeout: 5000               # 等待超时(ms)

  # 第三方API调用限流
  third_party:
    openai:
      rpm: 30                  # 保守设置，实际上限60RPM的50%
      tpm: 50000               # 保守设置
      retry_attempts: 3
      retry_backoff: "exponential"  # 指数退避
    tiktok:
      rpm: 50                  # 视具体套餐而定
      retry_attempts: 2
```

### 4.2 用户级限流

```yaml
user_level:
  # 免费用户
  free_tier:
    daily_requests: 100        # 每日限制
    hourly_requests: 20        # 每小时限制
    concurrent_requests: 3    # 并发限制
    
  # 付费用户
  pro_tier:
    daily_requests: 1000
    hourly_requests: 100
    concurrent_requests: 10
```

### 4.3 功能模块限流

| 模块 | QPS上限 | 备注 |
|------|--------|------|
| 产品搜索 | 10/s | 允许突发到30/s |
| 竞品分析 | 5/s | CPU密集型，保守设置 |
| 数据同步 | 2/s | TikTok API限制 |
| 报告生成 | 1/s | 成本较高 |

## 五、降级和熔断策略

### 5.1 多级降级机制

```
┌─────────────────────────────────────────────────────┐
│                     请求入口                         │
└─────────────────────┬───────────────────────────────┘
                      ▼
        ┌──────────────────────────┐
        │   一级：限流（令牌桶）    │ ← 拒绝请求，返回429
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │   二级：熔断器（Hystrix） │ ← 快速失败，保护下游
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │   三级：功能降级          │ ← 简化逻辑/返回缓存
        └────────────┬─────────────┘
                     ▼
        ┌──────────────────────────┐
        │   四级：服务降级          │ ← 返回友好提示
        └──────────────────────────┘
```

### 5.2 熔断器配置

```yaml
circuit_breaker:
  # 熔断触发条件
  error_threshold: 50%         # 错误率超过50%触发
  volume_threshold: 10         # 最小请求数
  
  # 熔断时长
  sleep_window: 30s            # 熔断持续时间
  half_open_requests: 3        # 半开状态探测请求数
  
  # 恢复条件
  success_threshold: 2          # 连续成功2次则恢复
```

### 5.3 功能降级策略

| 降级级别 | 触发条件 | 降级动作 | 用户感知 |
|---------|---------|---------|---------|
| **L1** | 响应>2s | 返回缓存数据 | 无感知 |
| **L2** | 响应>5s | 简化分析维度 | 轻微延迟 |
| **L3** | 熔断触发 | 返回历史报告 | 功能受限提示 |
| **L4** | 服务不可用 | 显示友好错误页 | 暂时不可用 |

### 5.4 重试机制

```yaml
retry:
  strategy: "exponential_backoff"
  attempts: 3
  delays: [1000, 2000, 4000]   # ms，指数退避
  jitter: true                  # 添加随机抖动避免惊群
  
  # 仅对幂等操作重试
  retryable_methods: ["GET", "POST"]
  non_retryable: ["DELETE", "PUT"]
```

## 七、监控指标

### 7.1 核心指标

| 指标 | 告警阈值 | 处理动作 |
|------|---------|---------|
| API错误率 | >5% | 触发熔断检查 |
| P99延迟 | >3s | 启动降级 |
| 限流拒绝率 | >10% | 扩容或调参 |
| 第三方API成功率 | <95% | 检查配额/切换节点 |

### 7.2 监控面板

建议使用 Grafana + Prometheus 搭建监控：
- 实时QPS监控
- 限流事件统计
- 熔断器状态
- 用户请求分布

| 策略 | 节省比例 | 实施难度 |
|------|---------|---------|
| 请求合并（批量处理） | 30-40% | 中 |
| 结果缓存（5分钟TTL） | 50-60% | 低 |
| 智能降级（缓存优先） | 20-30% | 中 |
